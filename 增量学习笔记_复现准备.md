## 1. 模型后端架构的精确理解

### 1.1 后端注册与选择逻辑

`backend_config.py` 中的关键逻辑：

```python
# optimizer 只允许两种后端
def set_optimizer_backend(backend: str) -> None:
    if backend not in {"openai_chat", "claude_chat"}:
        raise ValueError(f"Unsupported optimizer backend: {backend!r}")

# target 允许五种后端
def set_target_backend(backend: str) -> None:
    if backend not in {"openai_chat", "claude_chat", "qwen_chat", "codex_exec", "claude_code_exec"}:
        raise ValueError(...)
```

**关键洞察**：optimizer 的 backend 限制比 target 严格。optimizer **必须是 chat model**（需要结构化输出 JSON），而 target 可以是 exec backend（Codex CLI / Claude Code）。

### 1.2 `openai_chat` 的通用性

`openai_chat` backend 不是特指 OpenAI 的 API，而是**任何 OpenAI 兼容的 endpoint**。这意味着：

```python
# 这些全部映射到同一个 openai_chat backend：
# - https://api.openai.com/v1          (OpenAI)
# - https://your-azure.openai.azure.com (Azure OpenAI)
# - http://localhost:8000/v1           (vLLM 本地)
# - http://localhost:11434/v1          (ollama)
# - https://api.together.xyz/v1        (Together AI)
```

**切换只需要改环境变量**：
```bash
export SKILLOPT_AUTH_MODE=openai
export OPENAI_API_KEY=EMPTY
export OPENAI_BASE_URL=http://localhost:8000/v1
```

代码内部通过 `openai.OpenAI(base_url=..., api_key=...)` 统一调用。

### 1.3 `qwen_chat` vs `openai_chat` 的区别

```python
# qwen_chat backend
BASE_URL = os.environ.get("QWEN_CHAT_BASE_URL", "http://localhost:8000/v1")
TARGET_DEPLOYMENT = os.environ.get("TARGET_DEPLOYMENT", "Qwen/Qwen3.5-4B")

# openai_chat backend（通过 config）
model:
  target_backend: openai_chat
  target: Qwen/Qwen2.5-7B-Instruct
  openai_base_url: http://localhost:8001/v1
```

**差异**：`qwen_chat` 是专门为 Qwen 模型优化的 backend（可能支持 thinking 模式等 Qwen 特有功能），而 `openai_chat` 是通用的。对于 vLLM 部署，两者效果相同，但 `openai_chat` 的配置更灵活（可以通过 config 的 `openai_base_url` 指定不同 endpoint）。

### 1.4 Train.py 的 Legacy CLI 映射

`scripts/train.py` 有完整的 legacy flat CLI → structured config 映射表（`_LEGACY_TO_STRUCTURED`），包含 70+ 个字段。这意味着：

```bash
# 这两种写法完全等价：
python scripts/train.py --config configs/livemath/default.yaml --batch_size 20
python scripts/train.py --config configs/livemath/default.yaml --cfg-options train.batch_size=20
```

**深层含义**：项目经历了从 flat config 到 structured config 的迁移，但保留了完整的向后兼容。这是成熟工程项目的标志。

## 2. Dual Deployment 的技术实现细节

### 2.1 实际代码中的分离

`backend_config.py` 中，optimizer 和 target 有**完全独立的**环境变量路径：

```python
# Optimizer 的环境变量
OPTIMIZER_BACKEND = os.environ.get("OPTIMIZER_BACKEND", "openai_chat")
# 通过 set_optimizer_backend() 设置

# Target 的环境变量
TARGET_BACKEND = os.environ.get("TARGET_BACKEND", "openai_chat")
# 通过 set_target_backend() 设置
```

而 `train.py` 的 `load_config()` 中：
```python
if backend in {"qwen", "qwen_chat"}:
    flat.setdefault("optimizer_backend", "openai_chat")  # optimizer 必须是 chat
    flat.setdefault("target_backend", "qwen_chat")        # target 可以是 qwen
```

### 2.2 完全独立的 endpoint

Config 中 optimizer 和 target 有各自独立的 endpoint 配置：

```yaml
model:
  # Optimizer 的 Azure 配置
  optimizer_azure_openai_endpoint: ""
  optimizer_azure_openai_api_key: ""
  optimizer_azure_openai_auth_mode: ""
  
  # Target 的 Azure 配置
  target_azure_openai_endpoint: ""
  target_azure_openai_api_key: ""
  target_azure_openai_auth_mode: ""
  
  # 通用 OpenAI 兼容配置
  openai_base_url: ""          # 用于 optimizer（如果 optimizer_backend=openai_chat）
  openai_api_key: ""
```

**这意味着你可以让 optimizer 走 Azure OpenAI（GPT-5.5），target 走本地 vLLM（Qwen2.5-7B），完全混合使用。**

### 2.3 Exec Backend 的特殊性

`codex_exec` 和 `claude_code_exec` 是 **target-only** 的 backend：
- 它们不通过 OpenAI API 调用，而是**启动子进程**运行 CLI 工具
- CLI 工具内部自己处理 API 认证（不经过 SkillOpt-Lite 的 auth 系统）
- SkillOpt-Lite 只负责：准备 workspace（skill.md、task.md）→ 启动子进程 → 读取输出 → 解析轨迹

## 3. Optimizer 本地化的完整路径

### 3.1 为什么 Optimizer 本地化比 Target 更有价值

| 组件 | 调用频率（per step, batch=40） | 单次调用成本 | 本地化 ROI |
|------|-----------------------------|-------------|-----------|
| Optimizer (reflect) | ~10 calls | 高（长 prompt + 长输出） | **极高** |
| Optimizer (aggregate) | ~5 calls | 中 | 高 |
| Optimizer (ranking) | 1 call | 中 | 中 |
| Target (rollout) | 40 calls | 中 | 中 |
| Target (evaluate) | ~35 calls | 中 | 中 |

**Optimizer 本地化的 ROI 最高**，因为：
1. Optimizer 的调用量虽然少，但每次处理的文本量大（12KB trajectory + skill）
2. 如果用 GPT-5.5 做 optimizer，每次调用成本较高
3. 14B 本地模型足以处理这种分析任务

### 3.2 实际可行的本地化方案

**方案 A：完全本地（推荐用于复现）**
```
Optimizer: vLLM serving Qwen2.5-14B (port 8000)
Target:    vLLM serving Qwen2.5-7B  (port 8001)
```

**方案 B：Hybrid（推荐用于追求质量）**
```
Optimizer: Azure OpenAI GPT-5.4-mini（便宜，调用量少）
Target:    vLLM serving Qwen2.5-7B  (port 8001)
```

**方案 C：完全 cloud（原始论文方案）**
```
Optimizer: Azure OpenAI GPT-5.5
Target:    Azure OpenAI GPT-5.4-nano
```

### 3.3 小模型的局限性

用 7B/14B 模型替代 GPT-5.5 作为 optimizer 时，预期会遇到：

| 局限 | 表现 | 缓解策略 |
|------|------|---------|
| JSON 输出格式不稳定 | extract_json() 解析失败 | 在 prompt 中加强格式要求；增大 max_retries |
| 分析深度不足 | failure_summary 太浅 | 用 14B 而非 7B；在 prompt 中要求 "think step by step" |
| 合并 patch 质量差 | merge 阶段丢失重要 edit | 减小 merge_batch_size；增加 merge 的 retry |
| 排序不准确 | top-L 选择不当 | 用 simpler ranking 策略（support_count 优先） |

## 4. 系统瓶颈的精确量化

### 4.1 真正的时间消耗在哪里？

从 `step_record.json` 的 `timing` 字段可以看到：

| 阶段 | 典型耗时（cloud API） | 典型耗时（本地 vLLM 14B） | 瓶颈原因 |
|------|---------------------|------------------------|---------|
| Rollout | 5-30s | 30-120s | API RTT / vLLM 生成速度 |
| Reflect | 10-60s | 30-120s | LLM 处理长 prompt |
| Aggregate | 2-10s | 5-20s | LLM 合并 |
| Select | 2-5s | 2-5s | 纯 Python / 小 LLM 调用 |
| Evaluate | 5-30s | 20-60s | 同 Rollout |
| **总计** | **25-135s** | **90-330s** | — |

**本地 vLLM 的主要瓶颈是 token 生成速度**，而不是 API RTT。

### 4.2 vLLM 吞吐估算

对于 Qwen2.5-14B，单卡 4090：
- **Prefill 阶段**（处理输入）：~500-1000 tok/s
- **Decode 阶段**（生成输出）：~20-50 tok/s

Reflect 阶段的典型 prompt：
- 输入：~12KB trajectory text + skill（~3000 tokens）
- 输出：~500-2000 tokens（JSON patch）
- 总耗时：prefill 3-6s + decode 10-40s = **15-50s per call**

Rollout 阶段的典型调用：
- 输入：~2000 tokens（skill + task）
- 输出：~500-3000 tokens（agent response）
- 总耗时：prefill 1-2s + decode 10-60s = **15-65s per call**

### 4.3 优化方向

| 优化 | 效果 | 实现难度 |
|------|------|---------|
| 增大 `max-model-len` 的 chunked prefill | +20-30% throughput | 低（vLLM 参数） |
| 开启 `enable-chunked-prefill` | +10-20% throughput | 低（默认已开启） |
| 减小 `max_tokens`（rollout） | -30% 单条耗时 | 中（可能影响质量） |
| 增大 `tensor-parallel-size` | 线性加速（如果模型放得下） | 中（需要多卡） |
| 量化（AWQ/GPTQ） | 2-4x 速度提升 | 中（需要量化脚本） |
| speculative decoding | +30-50% decode 速度 | 高（vLLM 配置） |

## 5. Config 系统的继承与覆盖机制

### 5.1 `_base_` 继承的实际行为

```yaml
# configs/livemath/default.yaml
_base_: ../_base_/default.yaml

train:
  batch_size: 40  # 覆盖 base 的 40（相同值，但展示机制）
  accumulation: 1  # 覆盖 base 的 1
```

`config.py` 中的 `_load_config_with_inheritance()` 会：
1. 递归加载 `_base_` 链（支持多级继承）
2. Deep merge（子 config 的 dict 会覆盖父 config 的同名 key）
3. 处理 `delete_` 前缀（可以删除父 config 的 key）

### 5.2 Deep Merge vs Shallow Merge

```python
# base config
model:
  azure_openai:
    endpoint: "https://..."
    api_version: "2024-12-01-preview"

# child config（如果用 shallow merge）
model:
  azure_openai:
    endpoint: "https://different..."  # 整个 azure_openai 被替换，api_version 丢失！

# child config（如果用 deep merge）
model:
  azure_openai:
    endpoint: "https://different..."  # 只有 endpoint 被替换，api_version 保留
```

**项目用的是 deep merge**，所以子 config 只需要覆盖变化的字段。

### 5.3 `flatten_config()` 的转换

Structured config（YAML with sections）→ Flat dict（trainer 期望的格式）：

```yaml
# structured
model:
  optimizer: gpt-5.5
  target: gpt-5.5

train:
  batch_size: 40
```

→
```python
# flat
{
  "model_optimizer": "gpt-5.5",
  "model_target": "gpt-5.5",
  "train_batch_size": 40,
  ...
}
```

**注意**：`flatten_config()` 把 `section.key` 变成 `section_key`（下划线连接）。这是 trainer 和 adapter 期望的格式。

## 6. Resume 机制的边界条件

### 6.1 正常 Resume 流程

```
中断 → 重启 → 读取 runtime_state.json → 恢复 last_completed_step + 1
         ↓
    跳过已完成的 rollout（results.jsonl 存在）
    跳过已完成的 minibatch reflect（patches/*.json 存在）
    跳过已完成的 slow_update / meta_skill（done marker 存在）
```

### 6.2 边界情况：中断在 apply_patch 之后、evaluate 之前

如果训练被 `kill -9` 在以下位置：
```
apply_patch() 完成 → current_skill 已被修改
    ↓
[中断]
    ↓
evaluate() 还没跑 → candidate 没有被 gate 决策
```

**重启后的状态**：
- `current_skill` 已经被修改（但 gate 没 accept）
- `runtime_state.json` 的 `last_completed_step` 还是上一步
- 重启后会从这一步重新开始，但 `current_skill` 已经是修改后的版本

**潜在问题**：如果重启后，这一步的 rollout 使用了修改后的 skill，但 gate 可能 reject 这个修改，导致 current_skill 回滚。但 `current_origin` 会记录这个混乱的过程。

### 6.3 实践建议

```bash
# 如果怀疑 resume 状态不一致：
# 1. 检查 history.json 的最后一步
tail -c 1000 outputs/*/history.json | python -m json.tool

# 2. 检查 current_skill 和 best_skill 的差异
diff outputs/*/skills/skill_v$(jq -r '.[-1].step' outputs/*/history.json).md outputs/*/best_skill.md

# 3. 如果不一致，手动回滚
cp outputs/*/skills/skill_v$(jq -r '.[-1].step' outputs/*/history.json).md outputs/*/current_skill.md
```

## 7. 每个 Benchmark 的实际特征

### 7.1 LiveMathematicianBench（最简单，推荐先做）

| 特征 | 值 |
|------|-----|
| 任务类型 | 数学多选题（A-E） |
| 轮数 | 单轮（max_turns=1） |
| 数据量 | train=36, val=35, test=106 |
| Exec timeout | 300s |
| Workers | 64 |
| 特殊功能 | `--enable_tools`（sympy）、`--enable_router`（skill router） |
| 数据完整性 | ✅ 完整（不需要额外 corpora） |

**推荐作为第一个复现目标**：单轮、快速 eval、数据完整、signal 清晰。

### 7.2 SpreadsheetBench（最复杂）

| 特征 | 值 |
|------|-----|
| 任务类型 | 三类：cell_level、sheet_level、other |
| 轮数 | 最多 30 turns（ReAct） |
| 数据量 | train=80, val=39, test=162 |
| Exec timeout | 600s |
| Workers | 24 |
| 特殊功能 | Code generation + ReAct 两种模式 |
| 数据完整性 | ⚠️ 需要额外下载 spreadsheet 文件 |

**核心挑战**：
- 单条任务可能跑 5-10 分钟
- 需要 pandas/openpyxl 执行环境
- Failure type 多样（pandas-clobber、A1-confusion、hardcoded-ranges 等）

### 7.3 ALFWorld（最难，需要 simulator）

| 特征 | 值 |
|------|-----|
| 任务类型 | 具身 AI（ household tasks） |
| 轮数 | 最多 50 turns |
| 数据量 | train=200, val=140, test=134 |
| Exec timeout | 120s |
| Workers | 8 |
| 特殊功能 | 需要 ALFWorld simulator |
| 数据完整性 | ⚠️ 需要 `pip install alfworld[full]` + `alfworld-download` |

**核心挑战**：
- 需要完整的 simulator 环境
- 多轮交互，状态复杂
- 并发 rollout 可能互相干扰 simulator 状态

### 7.4 SearchQA / DocVQA / OfficeQA

| Benchmark | 特点 | 数据完整性 |
|-----------|------|-----------|
| SearchQA | 开放域 QA，需要搜索上下文 | ⚠️ 需要额外下载 corpora |
| DocVQA | 文档图像 QA（多模态） | ⚠️ 需要下载图像 |
| OfficeQA | 长文档 QA + tool use | ✅ 完整 |

**DocVQA 的特殊性**：需要 GPU 做图像推理（如果 target 是多模态模型）。但 SkillOpt-Lite 的 target 可以是纯文本模型，图像处理在 env 的 rollout 中完成。

## 8. Skill 作为 "Trainable Parameter" 的深层含义

### 8.1 Skill 不是 Prompt

这是最容易混淆的点：

| | Prompt | Skill (skill.md) |
|--|--------|-----------------|
| 生命周期 | 单次查询 | 跨多个 episode |
| 结构 | 通常是一段文本 | Markdown 文档（有章节、列表、代码块） |
| 修改方式 | 每次查询可以不同 | 训练过程中迭代修改 |
| 作用范围 | 单次 LLM 调用 | Agent 的完整行为规范 |
| 存储 | 内存中 | 文件系统（`skill.md`） |

**Skill.md 本质上是 Agent 的 "firmware"**——它定义了 Agent 的工具调用策略、错误恢复流程、领域知识。而 prompt 只是单次查询的 "指令"。

### 8.2 Skill 优化的 "梯度信号"

传统 ML：
```
Forward pass → Loss → Backward pass → Gradient → Weight update
```

SkillOpt-Lite：
```
Rollout → Score (hard/soft) → LLM Analyst → Edit operations → Apply → Gate
```

**"梯度" 是 LLM 对 trajectory 的分析结果**。这不是数学上的梯度，而是语义上的 "改进建议"。这种 "梯度" 的噪声比反向传播大得多，所以 gate 机制更加重要。

### 8.3 Edit 操作的语义

四种 edit 操作的设计哲学：

| 操作 | 语义 | 适用场景 |
|------|------|---------|
| `append` | 在 skill 末尾添加新内容 | 新增策略、补充注意事项 |
| `insert_after` | 在指定文本后插入 | 在相关章节后添加细节 |
| `replace` | 替换现有内容 | 修正错误描述 |
| `delete` | 删除内容 | 清理冗余或过时信息 |

**设计原则**：edit 操作应该是**原子的、可合并的、可回滚的**。这要求 skill.md 有清晰的结构（headings、sections），否则 edit 的 target 会不明确。

## 9. 实验可复现性的真正挑战

### 9.1 已控制的因素

| 因素 | 控制方式 |
|------|---------|
| 数据 split | 固定 seed=42，预生成 split |
| 模型 temperature | 固定为 0（或接近 0） |
| 代码 | 版本控制（git） |
| Random seed | 全局 seed 参数 |

### 9.2 未控制的因素（真正的挑战）

| 因素 | 影响 | 严重程度 |
|------|------|---------|
| LLM 输出的非确定性 | 即使 temperature=0，不同时间调用可能返回不同结果 | **高** |
| API rate limit 导致的时序差异 | ThreadPoolExecutor 的任务完成顺序不确定 | 中 |
| Network latency 波动 | Rollout 时间不同，可能导致不同的随机采样 | 低 |
| vLLM 的调度不确定性 | 相同 prompt 在不同时间可能有微小差异 | 中 |

### 9.3 缓解策略

```python
# 1. 对 LLM 输出做 hash，检测重复
skill_hash(skill_content)  # 用于 selection cache

# 2. Rollout 结果持久化，resume 时跳过
if os.path.exists(results_path):
    skip_this_task()

# 3. 多次实验取平均（论文没做，但复现时建议做）
for seed in [42, 43, 44]:
    run_experiment(seed)
```

### 9.4 本地 vLLM 的可复现性优势

本地 vLLM 比 cloud API **更容易保证可复现性**：
- 模型权重固定（同一版本）
-  deterministic CUDA kernels（如果设置 `--enforce-eager`）
- 无网络波动

但 vLLM 的 speculative decoding 和 continuous batching 可能引入非确定性。可以用 `--enforce-eager` 禁用 CUDA graph 来强制确定性。

## 10. HarnessOpt 的工程复杂度

### 10.1 两种编辑对象的本质区别

| | Skill Edit | Harness Edit |
|--|-----------|-------------|
| 目标 | skill.md（Markdown 文本） | rollout.py, react_agent.py 等（Python 代码） |
| Edit 类型 | append/insert_after/replace/delete | 传统的 code diff（patch） |
| 验证方式 | Selection set score | Selection set score（相同） |
| 回滚方式 | 恢复到之前的 skill.md | git revert（因为有 git tag） |
| 风险 | 文本语义错误 | 代码语法错误 → Agent 直接 crash |

### 10.2 HarnessOpt 的安全机制

从 `harnessopt-loop.prompt.md` 看到的关键设计：

```yaml
# Allowlist：允许编辑的文件
allowlist:
  - rollout.py
  - react_agent.py
  - codegen_agent.py
  - executor.py
  - recalc_harness.py
  - adapter.py

# Denylist：禁止编辑的文件
denylist:
  - evaluator.py      # 不能改评分逻辑！
  - dataloader.py     # 不能改数据加载！
  - configs/          # 不能改配置！
  - prompts/*.md      # 不能改 prompt！
  - skills/initial.md # 不能改初始 skill！
```

**设计哲学**：HarnessOpt 只能改 Agent 的**行为逻辑**，不能改**评估标准**或**数据**。否则会出现 "通过改评分标准来提高分数" 的作弊行为。

### 10.3 Diff Guard 机制

```bash
# 每次 harness edit 后自动检查
git diff --name-only HEAD | grep -vE '^(rollout|react_agent|codegen_agent|executor|recalc_harness|adapter)\.py$'
# 如果有 denylist 文件被修改 → 自动 reject
```

这是 HarnessOpt 的核心安全机制，防止 coding agent "走火入魔"。

## 11. 面试/技术讨论中的高价值洞察

### 11.1 "Skill 优化和 Meta Learning 的关系"

SkillOpt-Lite 的 slow update + meta skill 可以看作是一种 **meta-learning**：
- Slow update：跨 task 的 meta-knowledge（"在表格任务中，先确认列名"）
- Meta skill：跨 optimization 过程的 meta-knowledge（"这个环境下，高 support_count 的 edit 更可靠"）

**区别**：传统 meta-learning（如 MAML）通过梯度更新 meta-parameters；SkillOpt-Lite 通过 LLM 生成文本记忆。

### 11.2 "Why not RL？"

常见追问。核心答案：
1. **Reward 稀疏**：Agent 任务的 reward 是二元的（成功/失败），大部分轨迹都是失败的
2. **Credit assignment 困难**：Agent 的 30 轮交互中，哪一步导致了最终的成功/失败？
3. **Sample efficiency**：RL 需要大量探索，SkillOpt-Lite 的验证门控提供了更高效的 "exploitation"
4. **可解释性**：Skill edit 是人类可读的，RL policy 不是

**但 RL 有优势**：可以学习连续的、渐进的行为调整，而 skill edit 是离散的、跳跃式的。两者结合是一个研究方向。

### 11.3 "为什么不用 gradient-based 的 prompt optimization？"

如 OPRO、PromptAgent 等方法用 LLM 生成 prompt，但通常是 **一次性优化**（single-shot）。SkillOpt-Lite 的关键创新是 **闭环迭代**：
1. 生成 edit
2. 验证效果
3. 决定是否保留
4. 基于反馈生成下一轮 edit

这种 "generate → validate → iterate" 的循环比 single-shot 更稳定。

### 11.4 "Protected Region 和 Attention Mask 的类比"

NN 中的 attention mask 可以阻止信息流动。SkillOpt-Lite 的 protected region 通过 `apply_patch()` 中的检查来阻止 edit 流动：

```python
# skill.py 中的 apply_edit()
if _is_in_protected_region(skill_content, edit.target):
    return edit.copy(update={"status": "skipped_protected"})
```

这类似于 transformer 中的 masked attention——某些位置的信息流被显式阻止。

### 11.5 "Step Buffer 和 Context Window 的关系"

Step buffer 记录当前 epoch 内的失败模式，注入到 analyst 的 prompt 中。这有一个**隐含的约束**：

```
prompt 总长度 = system_prompt + skill_content + trajectory_text + step_buffer + ...
```

如果 step_buffer 太长，会挤占 trajectory_text 的空间。当前实现没有动态调整 step_buffer 大小的机制，这是一个潜在的工程问题。

### 11.6 "Hash-Based Cache 的精确语义"

```python
def skill_hash(skill_content: str) -> str:
    return hashlib.sha256(skill_content.encode()).hexdigest()[:16]
```

Cache 的 key 是 skill 内容的 SHA256 前 16 字符。这意味着：
- 即使 skill 只改了一个字符，hash 也会变，cache miss
- 如果 slow_update force-inject 了内容，但 skill 的其他部分没变，hash 会变吗？→ 会变，因为整个 skill 字符串变了
- 但 selection cache 在 `history.json` 中重建时会看到新的 hash

### 11.7 "Coding-Agent-Driven 的深层价值"

表面上，coding agent 只是 "更方便的用户界面"。但深层价值是：

1. **环境感知**：Coding agent 能看到项目结构、文件内容、git 状态，能做出 context-aware 的决策
2. **工具使用**：Coding agent 能运行命令、安装依赖、修改代码，比纯脚本更灵活
3. **人机协作**：HITL 模式通过文件 IPC，用户和 coding agent 能交替工作
4. **可移植性**：Slash command 是 markdown 文件，可以在 VS Code / Codex CLI / Claude Code 之间迁移
