## 1. 算力评估与模型选型

### 1.1 8× RTX 4090 能力边界

| 指标 | 数值 |
|------|------|
| 单卡显存 | 24 GB |
| 总显存 | 192 GB |
| FP16 算力 | ~330 TFLOPS（total） |
| 典型 vLLM 吞吐（7B, TP=1） | ~50-100 tok/s |
| 典型 vLLM 吞吐（14B, TP=1） | ~20-50 tok/s |
| 典型 vLLM 吞吐（32B, TP=1, 4-bit） | ~10-25 tok/s |

**结论：8×4090 完全足够。** 关键瓶颈不是 GPU 算力，而是 **vLLM 的 token 生成吞吐** 和 **并发 API 调度的延迟**。

### 1.2 模型选型矩阵

| 角色 | 推荐模型 | VRAM (4-bit) | 理由 | 备选 |
|------|---------|-------------|------|------|
| **Optimizer** | Qwen2.5-14B-Instruct | ~10-14 GB | 14B 在 trajectory analysis 和 patch generation 上质量最好 | Qwen2.5-7B（省显存，质量略降） |
| **Target (简单任务)** | Qwen2.5-7B-Instruct | ~5-8 GB | LiveMath/SearchQA 是单轮 QA，7B 足够 | Qwen2.5-4B（更快但质量明显下降） |
| **Target (复杂任务)** | Qwen2.5-14B-Instruct | ~10-14 GB | SpreadsheetBench 需要代码生成和多轮推理 | Qwen2.5-7B + 更多 turns |
| **Target (最大质量)** | Qwen2.5-32B-Instruct (4-bit) | ~18-22 GB | SpreadsheetBench  hardest cases | 不够时降级到 14B |

**推荐基线配置**：
```
Optimizer: Qwen/Qwen2.5-14B-Instruct  (vLLM, port 8000)
Target:    Qwen/Qwen2.5-7B-Instruct    (vLLM, port 8001)
```

这样两个 vLLM 实例共占 ~20 GB 显存，剩余 ~170 GB 可用于更大的模型或并行实验。

### 1.3 吞吐估算

以 LiveMath 为例（train=36, val=35, test=106, batch_size=40）：

| 阶段 | LLM 调用次数 | 单次调用延迟（14B, ~500 tok 输出） | 并行度 | 总耗时 |
|------|------------|--------------------------------|--------|--------|
| Rollout (train batch) | 40 | ~3-5s | 64 workers | ~30-60s |
| Reflect | ~10 (5 fail + 5 success × M=8) | ~5-10s | 16 workers | ~30-60s |
| Evaluate (selection, 35 items) | 35 | ~2-4s | 64 workers | ~20-40s |
| Slow Update (epoch 2+, 40 calls) | 40 | ~3-5s | 16 workers | ~30-60s |

**每 step 约 90-220 秒，每 epoch（~1 step）约 2-4 分钟，完整 4 epoch 训练约 10-15 分钟。**

这个估算基于 vLLM 本地推理。如果用 cloud API（Azure OpenAI），延迟更低（~1-2s），但需要 API key。

## 2. 复现策略总览

### 2.1 整体策略：分层递进

```
Phase 0: 环境搭建（vLLM + 数据 + 依赖）
    ↓
Phase 1: 单步 Smoke Test（验证 pipeline 能跑通）
    ↓
Phase 2: LiveMath 完整 Loop（最简单的 benchmark，快速迭代）
    ↓
Phase 3: SpreadsheetBench 完整 Loop（最复杂的 benchmark，验证 harness 能力）
    ↓
Phase 4: 消融实验（验证每个组件的贡献）
    ↓
Phase 5: HarnessOpt 探索（联合优化 skill + harness code）
```

### 2.2 关键决策

| 决策 | 选择 | 理由 |
|------|------|------|
| Optimizer 模型 | Qwen2.5-14B | 14B 在分析能力和速度之间最佳平衡 |
| Target 模型 | Qwen2.5-7B → 14B | 7B 足够简单任务；复杂任务升级 14B |
| 是否用 cloud API 做 optimizer | 否 | 目标是完全本地复现 |
| 是否复现论文精确数字 | 否 | 用不同模型，目标是验证 pipeline 有效性 |
| 优先哪个 benchmark | LiveMath | 单轮、fast eval、clean signal |
| 训练配置 | 缩小版 | batch_size=20（vs 40）, num_epochs=2（vs 4）先验证 |

### 2.3 与原始论文的关键差异

| 维度 | 原始论文 | 本复现 |
|------|---------|--------|
| Optimizer | GPT-5.5 | Qwen2.5-14B |
| Target | GPT-5.5 / GPT-5.4-nano | Qwen2.5-7B / 14B |
| batch_size | 40 | 20（先缩小验证） |
| num_epochs | 4 | 2（先缩小验证） |
| 数据 | 官方 HuggingFace | 同一来源，相同 split |
| 评估指标 | hard / soft | 同上 |

**预期**：分数不会超过论文，但应该能看到 **skill 优化的正向效果**（score 随 step 上升，gate accept 比例合理，slow update 有内容变化）。

## 3. Phase 0：环境搭建（Day 1）

### 3.1 硬件准备

```bash
# 确认 GPU 可见
nvidia-smi

# 确认驱动版本 >= 525（vLLM 要求）
nvidia-smi --query-gpu=driver_version --format=csv,noheader

# 确认 NCCL 可用
python -c "import torch; print(torch.cuda.nccl.version())"
```

### 3.2 软件安装

```bash
# 1. 创建 venv
python -m venv .venv && source .venv/bin/activate

# 2. 安装依赖
pip install -r requirements.txt

# 3. 额外安装 vLLM（用于本地模型 serving）
pip install vllm>=0.4.0

# 4. 验证 vLLM 能调用 GPU
python -c "from vllm import LLM; print('vLLM OK')"
```

### 3.3 配置本地 Auth

创建 `.env`：
```bash
# 使用 openai_compatible 模式，指向本地 vLLM
SKILLOPT_AUTH_MODE=openai
OPENAI_API_KEY=EMPTY
OPENAI_BASE_URL=http://localhost:8000/v1
```

修改 `env.sh` 或直接 export：
```bash
export SKILLOPT_AUTH_MODE=openai
export OPENAI_API_KEY=EMPTY
export OPENAI_BASE_URL=http://localhost:8000/v1
```

### 3.4 下载数据

```bash
# 下载所有 benchmark 数据（约几 GB）
python data/download.py

# 或按需下载
python data/download.py --config livemath
python data/download.py --config spreadsheetbench
```

**注意**：
- SearchQA / DocVQA / ALFWorld 需要额外 corpora（见 `docs/data.md`）
- SpreadsheetBench 需要额外的 spreadsheet 文件（从原始 repo 下载）
- 初次复现建议先做 **LiveMath**（数据完整，不需要额外 corpora）

### 3.5 启动 vLLM 服务

创建 `scripts/start_vllm.sh`：
```bash
#!/bin/bash
# Optimizer: Qwen2.5-14B on port 8000
# Target: Qwen2.5-7B on port 8001

# 先启动 optimizer
vllm serve Qwen/Qwen2.5-14B-Instruct \
    --tensor-parallel-size 1 \
    --port 8000 \
    --gpu-memory-utilization 0.85 \
    --max-model-len 32768 \
    --dtype half \
    --enable-chunked-prefill \
    --swap-space 4 &

# 再启动 target（如果需要）
vllm serve Qwen/Qwen2.5-7B-Instruct \
    --tensor-parallel-size 1 \
    --port 8001 \
    --gpu-memory-utilization 0.85 \
    --max-model-len 32768 \
    --dtype half \
    --enable-chunked-prefill \
    --swap-space 4 &

wait
```

### 3.6 验证 vLLM 服务

```bash
# 测试 optimizer
curl http://localhost:8000/v1/chat/completions \
    -H "Content-Type: application/json" \
```

## 4. Phase 1：单步 Smoke Test（Day 1-2）

### 4.1 目标

验证完整 pipeline 的**单步**能跑通：
1. Rollout：用初始 skill 跑 5 条轨迹
2. Reflect：minibatch reflection 能产出 patch
3. Aggregate：patch 能合并
4. Select：edit 能被排序
5. Update：skill 能被修改
6. Evaluate：gate 能决策

### 4.2 执行命令

```bash
# 先确保 vLLM 在运行
python scripts/eval_only.py \
    --config configs/livemath-local-vllm/default.yaml \
    --skill skillopt/envs/livemathematicianbench/skills/initial.md \
    --split train \
    --eval_limit 5 \
    --target_model Qwen/Qwen2.5-7B-Instruct \
    --reasoning_effort medium \
    --workers 4 \
    --out_root outputs/smoke_test
```

### 4.3 验证 checklist

- [ ] `results.jsonl` 有 5 条结果，每条有 `hard`、`soft`、`n_turns`
- [ ] `eval_summary.json` 存在且分数合理
- [ ] 如果跑 train.py，检查 `outputs/` 下是否有 `step_0001/` 目录
- [ ] `step_0001/step_record.json` 有 `action` 字段（accept/reject）
- [ ] `step_0001/candidate_skill.md` 和 `step_0001/merged_patch.json` 存在

### 4.4 常见问题排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| vLLM 返回 404 | model id 不匹配 | 检查 `/v1/models` 端点 |
| Rollout 全 0 分 | target model 输出格式错误 | 检查 env 的 parse 逻辑 |
| Reflect 阶段 JSON 解析失败 | 小模型输出格式不规范 | 增大 `max_analyst_rounds` 或换更大模型 |
| Gate 始终 reject | learning_rate 太大 | 减小到 2，或检查 selection set 分数 |

## 5. Phase 2：完整 Loop 复现 — LiveMath（Day 2-4）

### 5.1 目标

在 LiveMath 上跑完整训练 loop（2 epoch），验证：
1. Score 随 step 上升
2. Gate accept/reject 比例合理（~30-50% accept）
3. Slow Update 产生有意义的 protected region 内容
4. Meta Skill 在 epoch 边界被触发

### 5.2 实验配置

```bash
python scripts/train.py \
    --config configs/livemath-local-vllm/default.yaml \
    --num_epochs 2 \
    --batch_size 20 \
    --accumulation 1 \
    --minibatch_size 8 \
    --analyst_workers 8 \
    --learning_rate 4 \
    --min_learning_rate 2 \
    --lr_scheduler cosine \
    --use_slow_update true \
    --slow_update_samples 10 \
    --use_meta_skill true \
    --eval_test true \
    --seed 42
```

### 5.3 预期结果

| 指标 | 预期值 | 验证方式 |
|------|--------|---------|
| 基线 score (skill_init) | ~0.2-0.4 | `runtime_state.json` 的 `current_score` |
| 最终 score | > 基线 + 5-10% | `summary.json` 的 `test_hard` |
| Accept rate | 30-50% | `history.json` 统计 |
| Slow update 有内容 | protected region 非空 | `slow_update/epoch_*/slow_result.json` |
| Meta skill 有内容 | 非空 | `meta_skill/epoch_*/meta_skill_result.json` |

### 5.4 调试检查点

**如果 score 不上升**：
1. 检查 `history.json` 的每步 `action` — 如果全是 reject，说明 optimizer 生成的 edit 不好
2. 检查 `merged_patch.json` 的 edits 质量 — 是否在重复相同的 edit？
3. 检查 `selection_hard` 是否波动太大 — 可能是 selection set 太小
4. 尝试增大 `minibatch_size` 或换更大的 optimizer 模型

**如果 rollout 阶段全失败**：
1. 检查 target model 的输出格式 — Qwen2.5 是否遵循了 skill 中的格式要求？
2. 检查 `results.jsonl` 的 `fail_reason` — 是模型不配合还是 env 的 eval 逻辑有问题？
3. 用 `scripts/eval_only.py --eval_limit 1` 单条调试

## 6. Phase 3：完整 Loop 复现 — SpreadsheetBench（Day 4-7）

### 6.1 目标

在更复杂的 benchmark 上验证 pipeline 的鲁棒性：
1. Multi-turn ReAct agent（最多 30 turns）
2. Code generation + 执行
3. 更长的 exec_timeout（600s）

### 6.2 额外依赖

```bash
# SpreadsheetBench 需要额外的 spreadsheet 数据
# 从 https://github.com/RUCKBReasoning/SpreadsheetBench 下载
mkdir -p data/spreadsheetbench/spreadsheet
# 下载并解压到 data/spreadsheetbench/spreadsheet/
```

### 6.3 实验配置

```bash
python scripts/train.py \
    --config configs/spreadsheetbench-local-vllm/default.yaml \
    --num_epochs 2 \
    --batch_size 20 \
    --accumulation 1 \
    --minibatch_size 8 \
    --analyst_workers 8 \
    --learning_rate 4 \
    --use_slow_update true \
    --slow_update_samples 10 \
    --eval_test true \
    --seed 42
```

### 6.4 预期挑战

| 挑战 | 原因 | 应对 |
|------|------|------|
| 单条 rollout 很长（30+ turns） | Agent 需要多步推理 | 增大 `exec_timeout` 到 300-600s |
| 成本高 | 每条消耗很多 token | 缩小 batch_size 到 10-20 |
| 某些任务持续 timeout | Agent 陷入循环 | 在 env 的 rollout 中加 early stop |
| Target 7B 不够用 | 代码生成能力弱 | 升级到 Qwen2.5-14B |

## 7. Phase 4：消融实验与对比（Day 7-10）

### 7.1 消融矩阵

在 LiveMath 上跑以下配置，对比最终 test score：

| 实验 | 配置变更 | 预期影响 |
|------|---------|---------|
| Baseline | 原始 skill，不训练 | 最低分 |

## 8. Phase 5：HarnessOpt 探索（Day 10-14）

### 8.1 目标

在 SpreadsheetBench 上探索 **joint optimization of skill + harness code**：
1. 修改 `rollout.py`、`react_agent.py` 等代码
2. 用 `/harnessopt-loop` 的思路驱动
3. 验证联合优化能进一步提升效果

### 8.2 实现方式

HarnessOpt 的 loop 目前由 coding agent 驱动（slash-command prompt）。在本地复现时，有两种方式：

**方式 A：人工模拟 coding agent**
1. 读 `workspace/.skillopt/samples/failed/*.md`
2. 手动分析 failure pattern
3. 手动 edit `rollout.py` / `react_agent.py`
4. 跑 `eval_only.py` 验证
5. 决定 accept/reject

**方式 B：用 LLM 自动生成 harness edit**
1. 在 reflect 阶段，除了生成 skill edits，还生成 harness edit proposals
2. 修改 trainer 的 reflect/aggregate 阶段，产出两种类型的 patch
3. 在 update 阶段，分别应用到 skill.md 和 Python 文件

方式 B 更自动化，但实现量更大。建议先用方式 A 验证可行性，再考虑自动化。

## 10. 风险评估与应对

### 10.1 高风险

| 风险 | 影响 | 概率 | 应对 |
|------|------|------|------|
| 小模型 quality 不足 | optimizer 生成的 edit 质量差，score 不升 | **高** | 升级到 14B；如果还不行，考虑用 cloud API 做 optimizer（hybrid 方案） |
| vLLM 显存不足 | 模型加载失败 | 中 | 用 4-bit 量化（`--quantization awq`）；减小 `max-model-len` |
| SpreadsheetBench rollout 超时 | 训练无法完成 | 中 | 缩小 batch_size；减少 max_turns；增大 exec_timeout |
| 数据下载失败 | 无法开始训练 | 低 | 手动下载；用 HF datasets 库直接加载 |

### 10.2 中风险

| 风险 | 影响 | 概率 | 应对 |
|------|------|------|------|
| LiveMath eval 逻辑和 model 输出不匹配 | 全 0 分 | 中 | 仔细阅读 env adapter 的 parse 逻辑；用 GPT 先验证 |
| ThreadPoolExecutor 并发问题 | rollout 结果丢失 | 低 | 减少 workers；检查 results.jsonl 完整性 |
| Config 继承出错 | 参数不生效 | 中 | 用 `flatten_config()` 打印最终 config |

### 10.3 Hybrid 方案（如果纯本地不够）

如果 14B optimizer 质量确实不够，可以采用 hybrid：
```
Optimizer: GPT-5.4-mini (cloud API, 便宜)
Target: Qwen2.5-7B (local)
```

这样 optimizer 的质量接近论文，target 的成本接近本地。混合成本：
- Optimizer 调用量小（~10-20 calls per step），GPT-5.4-mini 很便宜
- Target 调用量大（~40 calls per step），本地推理免费
