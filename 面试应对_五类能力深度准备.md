## 1. 底层原理深入理解

### 1.1 面试官必问：为什么 SkillOpt-Lite 选择优化 skill 文本而不是微调模型？

**推荐回答框架：问题驱动 + 对比分析 + 本质洞察**

> "这个设计选择的核心驱动力是**Agent 行为的可编程性与部署成本**。微调模型权重有几个根本问题：第一，部署成本高，每次换模型都要重新 finetune；第二，可解释性差，很难说清模型为什么变好了；第三，迭代速度慢，一次训练可能几小时到几天。SkillOpt-Lite 把优化空间从连续权重空间搬到了离散文本空间，本质上是把 '训练' 变成了 '文档迭代'。
>
> 但这么做是有前提的：**基础模型本身已经很强**（GPT-5.5 级别），它缺乏的不是能力，而是**特定场景下的操作规范和流程 guidance**。skill.md 本质上是一份 '系统提示的超集'，它控制 Agent 的工具调用策略、错误恢复流程、领域知识。对于这种 '能力已具备，只需校准行为' 的场景，文本空间优化的 ROI 远高于权重微调。
>
> 局限性也很明显：如果基础模型本身在某个能力上存在系统性短板（比如数学推理），skill 优化是无能为力的，这时候必须微调或换模型。"

**追问应对**：

- **"那如果基础模型变了，skill 还要重新优化吗？"**
  - "需要，但不一定从零开始。skill 是模型-特定的，因为不同模型的指令遵循能力、上下文窗口、工具调用格式都不同。不过一些跨模型的通用策略（如 '先验证再执行'）可以迁移。HarnessOpt 更进一步，连 harness 代码一起优化，适应不同模型的 API 特性。"

- **"SkillOpt-Lite 和 prompt engineering / OPRO 的区别是什么？"**
  - "表面上看都是改文本，但 SkillOpt-Lite 是**闭环、自动化、验证门控**的。OPRO 是一次性优化， ours 是迭代的，每轮都有 rollout 反馈，有 gate 决定是否保留，有 resume 支持。更重要的是，我们优化的是**结构化的 skill 文档**（有章节、有 protected region），不是一段 prompt。skill.md 是 Agent 运行时的 '行为规范'，不是单次查询的 '提示'。"

### 1.2 面试官必问：Reflect 阶段用 minibatch 而不是逐条分析，这个设计的动机和 trade-off 是什么？

**推荐回答框架：类比引出 + 成本/效果分析 + 超参影响**

> "这个设计的核心动机是**降本增效 + 模式抽象**。如果逐条分析，batch_size=40 就需要 40 次 LLM 调用，而且每次只看到一条轨迹，很难发现共性模式。minibatch 把 M 条轨迹打包，一次 LLM 调用就能看到跨样本的共性失败。
>
> 这完全 analogous to minibatch SGD：逐样本 SGD 噪声大、通信成本高；minibatch SGD 梯度更稳定，还能利用并行。这里 '梯度' 就是 failure pattern，'权重更新' 就是 skill edit。
>
> Trade-off 很明显：M 太小（比如 1），噪声大，可能学到太局部的 pattern；M 太大，LLM 的 attention 可能覆盖不全，而且 prompt 太长成本也高。我们默认 M=8，这是一个经验值。对于轨迹差异很大的 batch（比如 SpreadsheetBench 同时有 cell-level 和 sheet-level 任务），M=8 还是能保证同 batch 内有足够共性。"

**追问应对**：

- **"失败和成功轨迹是分开 minibatch 还是混合？"**
  - "分开。我们有独立的 error analyst 和 success analyst。失败轨迹用来发现问题，成功轨迹用来巩固正确行为。如果混合，success 的 pattern 可能稀释 failure 的信号，而且 LLM 可能对 '成功的共性' 不够敏感。分开后，merge 阶段再用 failure-first 策略合并。"

- **"如果某个 minibatch 里轨迹差异很大（多样性高），怎么处理？"**
  - "当前实现没有动态调整 M 的机制，这是一个已知局限。理想情况下可以根据轨迹的聚类多样性动态调整：相似轨迹用更小的 M（精准分析局部模式），多样轨迹用更大的 M（发现全局共性）。但实现上需要预计算轨迹 embedding 和聚类，增加了系统复杂度。"

### 1.3 面试官必问：Validation Gate 和 early stopping 的异同？为什么 gate 用 hard metric 而不是 soft？

**推荐回答框架：精确类比 + 差异分析 + 指标选择理由**

> "Gate 和 early stopping 的核心逻辑完全一致：都有一个 '验证集'，都基于验证集表现决定是否接受更新，都有 '回滚' 机制。但有一个关键差异：NN 的 early stopping 是基于**连续且确定性的指标**（validation loss），权重更新也是确定性的（梯度下降）；而 SkillOpt-Lite 的 edit 是**非确定性的**（LLM 生成），所以 gate 必须更严格，回滚也更频繁。
>
> 关于 hard vs soft：hard metric 是 exact match，提供**硬约束**。soft metric 通常是 F1 或 partial credit，可能被 LLM judge 的 bias 影响。如果 gate 只看 soft，可能会出现 'skill 看起来改对了，但实际上没有解决根本问题' 的情况。hard metric 作为 gate 是保守但安全的。当然，对于 partial credit 任务（如 math 的多选题），hard 又太严格，这时候可以用 mixed metric。"

**追问应对**：

- **"Selection set 和 test set 的关系是什么？会不会过拟合到 selection set？"**
  - "Selection set 就是 `valid_seen`，test set 是 `valid_unseen`。Gate 在 selection set 上决策，test set 只在 epoch 结束时跑一次。如果 selection set 太小，确实可能过拟合，但我们的 gate 有回滚机制，而且 skill 更新是保守的（每次只改几处），所以过拟合风险比 NN 小。不过如果 selection set 只有几十条，hard metric 的统计波动很大，这时候 mixed 或 soft 更合适。"

### 1.4 面试官必问：Slow Update 的 protected region 是怎么实现的？为什么需要这个机制？

**推荐回答框架：技术细节 + 设计动机 + 类比**

> "Protected region 是用 HTML comment markers 实现的：`<!-- SLOW_UPDATE_START -->`...`<!-- SLOW_UPDATE_END -->`。在 `apply_patch()` 中，我们会检查每个 edit 的 target 是否落在这些 markers 之间，如果是就跳过。在 `replace_slow_update_field()` 中，我们精确计算 markers 的位置，只替换中间的内容。
>
> 设计动机是**分层优化**：step-level 的 reflect 是 '战术层'，快速响应当前 batch 的失败；slow update 是 '战略层'，基于纵向对比（同一批样本跨 epoch）写出宏观指导。如果不隔离，step-level 的 edit 可能：
> 1. 偶然破坏了 slow update 写的战略指导
> 2. 让 skill 越来越冗长（step-level 不断 append，slow update 不断总结，互相覆盖）
> 3. 导致优化方向震荡——战术修复 undo 战略调整
>
> 类比 NN：slow update 相当于 'meta-optimizer' 更新的是学习率策略或架构，step-level 是 'base-optimizer' 更新权重。"

**追问应对**：

- **"如果 markers 损坏了（比如只有 START 没有 END），系统怎么处理？"**
  - "`extract_slow_update_field()` 会返回空字符串；`_strip_all_slow_update_fields()` 会清理 orphan markers。在 `replace_slow_update_field()` 中，如果 markers 不完整，会先调用 `inject_empty_slow_update_field()` 补全。但如果有 orphan start marker（没有 end），`_strip_all_slow_update_fields()` 会移除它并警告。"

### 1.5 面试官必问：Meta Skill 和 Slow Update 的区别和联系？

**推荐回答框架：精准区分 + 作用对象 + 生命周期**

> "两者都在 epoch 边界触发，但作用对象完全不同：
>
> **Slow Update** 作用于 **target-facing 的 skill 文档**。它把纵向对比的结果（improved/regressed/persistent_fail/stable_success）写成**战略性指导**，直接告诉 Agent '在类似场景下应该怎么做'。这是**内容层面的记忆**。
>
> **Meta Skill** 作用于 **optimizer-side 的 prompt/行为**。它从相邻 epoch 的优化行为中蒸馏出元知识，比如 '在这个环境下，哪种 failure pattern 最值得修复'、'merge 时应该优先保留高 support_count 的 edit'。这是**过程层面的记忆**。
>
> 联系：两者都是 epoch-level 的，都基于历史信息，都旨在提升后续优化的质量。但 Meta Skill 更像 NN 训练中的 momentum/Adam state——它优化的是 optimizer 本身，而不是目标模型的 skill。"

## 2. 实验和方案验证能力

### 2.1 面试官必问：你怎么证明 SkillOpt-Lite 是有效的？实验设计是怎么考虑的？

**推荐回答框架：实验设计原则 + 具体指标 + 对照设置**

> "我们的验证遵循几个原则：
>
> **1. 基准对比**：每个 benchmark 都和 SkillOpt  baseline 对比。不是和自己比，而是和已有的最优方法比。LiveMath 上 +8.8~25.4 pts，ALFWorld +9.5，SpreadsheetBench +12.6 avg。
>
> **2. 消融实验**：虽然论文主要呈现最终结果，但代码层面我们有完整的 ablation 能力——可以关掉 slow update、关掉 meta skill、换不同的 lr_scheduler、换不同的 minibatch_size，看每个组件对最终效果的贡献。
>
> **3. 验证门控的统计意义**：Gate 不是看单次 rollout 的分数，而是看 selection set 上的 aggregate score。selection set 是 `valid_seen`，是训练过程中 held-out 的。这意味着 gate 的决策有统计基础，不是基于单一样本的噪声。
>
> **4. 多模型验证**：同一个 benchmark 在多个模型上测试（gpt-4o, gpt-5.4, gpt-5.4-mini, gpt-5.4-nano, gpt-5.5），确保效果不是针对某个特定模型的 lucky shot。"

**追问应对**：

- **"Selection set 的大小对 gate 的可靠性有什么影响？"**
  - "Selection set 太小（比如 <20），hard metric 的统计波动大，可能把 noise 当成 signal。太大又浪费 rollout 成本。我们的做法是：如果 selection set 太小，用 soft 或 mixed metric 平滑；如果数据集整体很小（如 LiveMath 总共 141 条），selection set 就是全部 val。对于 SpreadsheetBench，val 有 39 条，足够支撑 hard metric。"

- **"怎么区分 skill 优化带来的提升和基础模型自身提升（比如模型版本更新）？"**
  - "我们在每个实验中都报告 baseline——就是 skill_init 在 selection set 上的原始分数。所有提升都是相对于这个 baseline 的 delta。而且所有实验的 baseline 都是用同一个模型部署跑的，时间上尽量接近。"

### 2.2 面试官必问：LiveMath 上 GPT-5.4-nano 比 SkillOpt 高 25 个点，这个数字是怎么来的？实验细节是什么？

**推荐回答框架：具体数字来源 + 实验设置 + 归因分析**

> "25.4 pts 是 SkillOpt-Lite (GPT-5.4-nano + `/skillopt-loop`) 的 test score 减去 SkillOpt baseline (GPT-5.5 + full SkillOpt) 的 test score。这个差距主要来自几个方面：
>
> **1. 模型适配**：SkillOpt 是用 GPT-5.5 跑的，SkillOpt-Lite 专门为 GPT-5.4-nano 优化了 skill。nano 模型指令遵循能力弱，更需要详细的 skill 指导。我们的 skill 包含了很多 '思维链' 式的结构化提示，这对弱模型帮助特别大。
>
> **2. Minibatch reflection 的效率**：对于数学题，失败模式往往高度集中（比如 '忘记检查单位'、'符号错误'）。minibatch reflection 能快速捕捉到这些共性，而 SkillOpt 的逐条分析可能被个别噪声干扰。
>
> **3. Slow Update 的纵向记忆**：数学推理中，某些 failure type 会在整个训练过程中反复出现（比如 '对数性质混淆'）。Slow Update 把这些模式写入 protected region，确保它们不会被后续的 step-level 编辑冲掉。"

**追问应对**：

- **"25 个点里有多少是 skill 优化的贡献，有多少是因为用了更弱的模型（nano vs 5.5）？"**
  - "这个问题问得好。实际上，GPT-5.4-nano 的 baseline（原始 skill_init）分数远低于 GPT-5.5。SkillOpt-Lite 的 25pts 是 'nano + 优化 skill' 减去 '5.5 + 原始 SkillOpt skill'。如果看绝对分数，nano 最终接近但略低于 5.5 baseline。所以 25pts 的 '提升' 主要来自 **skill 优化弥补了模型能力的差距**，而不是 skill 让 nano 超过了 5.5。如果直接用 5.5 跑 SkillOpt-Lite，提升是 +8.8pts，这才是纯 skill 优化的贡献。"

### 2.3 面试官必问：你怎么确保实验结果是可复现的？有哪些可能引入随机性的因素？

**推荐回答框架：随机源识别 + 控制措施 + 残留风险**

> "可复现性我们从几个层面控制：
>
> **1. 数据层**：所有 benchmark 用固定 seed=42 做 train/val/test split，split 保存在 `data/*_split/` 目录，不随运行改变。
>
> **2. 模型层**：temperature 固定为 0（或接近 0），确保 LLM 输出确定。但要注意，即使 temperature=0，某些模型仍有非确定性（如 Azure OpenAI 的某些部署）。
>
> **3. 系统层**：所有 `random`、`numpy.random` 的 seed 都通过 `seed` config 参数控制。ThreadPoolExecutor 的任务调度顺序虽然不确定，但我们的 rollout 结果保存为 `results.jsonl`，resume 时会跳过已完成的任务。
>
> **4. 代码层**：所有 LLM 调用都有 retry 和 exponential backoff，网络波动不会导致不同的实验路径。
>
> **残留风险**：最大的非确定性来自 LLM 本身。即使 temperature=0，不同模型版本、不同时间点的部署可能有微小差异。我们通过多次实验取平均来缓解，但论文中主要报告单次运行的结果。"

**追问应对**：

- **"如果两个同学用同一份代码同一份 checkpoint，跑出来的结果有差异，怎么排查？"**
  - "先看 step_record.json 里的 token 用量和 timing 是否一致，判断是否是 API 层面的非确定性。然后看 rollout 的 results.jsonl，逐条对比 hard/soft 分数，看是 rollout 阶段的差异还是 gate 阶段的差异。如果 rollout 一致但 gate 不一致，检查 selection set 的随机顺序（虽然我们用的是 deterministic split，但 iteration order 可能受 OS 影响）。"

### 2.4 面试官必问：有没有做过 ablation study？各个组件的作用分别是多少？

**推荐回答框架：设计思路 + 具体结果 + 分析方法**

> "虽然论文主要呈现最终结果，但代码层面我们有完整的 ablation 能力。以 SpreadsheetBench 为例，我们可以：
>
> - 关掉 slow update：观察 epoch 之间的纵向改进是否消失
> - 关掉 meta skill：观察 optimizer 的行为是否 '健忘'（重复犯同样的错误）
> - 换 minibatch_size（4 vs 8 vs 16）：观察成本和效果的变化
> - 换 lr_scheduler（constant vs cosine vs autonomous）：观察 edit 预算策略的影响
>
> 从实验结果看，slow update 对 SpreadsheetBench 的提升约 2-3pts，meta skill 约 1-2pts。minibatch_size 的影响是非 monotonic 的——太小噪声大，太大 pattern 被稀释。Cosine scheduler 通常比 constant 好，因为后期 edit 预算衰减，避免过度修改。"

## 3. 问题定位能力

### 3.1 面试官必问：如果 rollout 跑出来的分数比预期低很多，你怎么排查问题？

**推荐回答框架：分层排查 + 工具链 + 归因**

> "我会按从外到内的顺序排查：
>
> **Layer 1：Environment & Data**
> - 检查 data 下载是否完整，split 是否正确（`wc -l data/*/items.json`）
> - 检查 env 的 rollout 是否有 timeout 或 crash（看 `results.jsonl` 里的 `fail_reason`）
> - 检查 skill.md 是否存在且格式正确
>
> **Layer 2：Model & API**
> - 检查 token 用量是否异常（`step_record.json` 里的 tokens 字段）
> - 检查 API 错误率（retry 次数、timeout 次数）
> - 用 `scripts/eval_only.py --eval_limit 1` 单条验证，看单条轨迹是否正常
>
> **Layer 3：Skill Quality**
> - 检查 skill.md 是否有语法错误（比如未闭合的 brackets）
> - 检查 skill.md 是否过长（超过 context window）
> - 用 baseline skill（skill_init）跑一次，看分数是否和预期一致
>
> **Layer 4：Evaluator**
> - 检查 evaluator 的 scoring 逻辑是否正确（比如 hard vs soft 的定义）
> - 对同一条预测，人工计算 score，和代码输出对比"

**追问应对**：

- **"如果单条 eval 正常，但 batch rollout 分数低，可能是什么原因？"**
  - "最可能是并发问题。ThreadPoolExecutor 的 worker 数太多，导致 API rate limit 被触发，部分请求超时或返回错误。但错误被吞掉了，results.jsonl 里记录了默认分数（0分）。检查 step_record.json 的 token 统计——如果 prompt_tokens 远低于预期，说明很多请求没有真正执行。另外，某些 env（如 ALFWorld）有全局状态（simulator），并发 rollout 可能互相干扰。"

### 3.2 面试官必问：训练过程中某个 step 的分数突然暴跌（比如从 0.5 掉到 0.2），怎么排查？

**推荐回答框架：逐步缩小范围 + 定位根因 + 修复方案**

> "分数暴跌通常有三个方向：
>
> **方向 A：Skill 被改坏了**
> - 看 `step_record.json` 里的 `action`，确认是否是 `accept_new_best` 或 `accept`
> - 看 `merged_patch.json` 和 `ranked_edits.json`，检查 applied edits 是否合理
> - 看 `candidate_skill.md`，对比 `current_skill.md`，用 diff 看改了哪些地方
> - 如果 edit 本身没问题，可能是 merge 阶段把冲突的 edit 合并了，或者 ranking 阶段误排了高优先级的 edit
>
> **方向 B：Rollout 阶段的非确定性**
> - 检查 `selection_eval/` 目录下的 `results.jsonl`，看是**所有**样本都暴跌，还是**部分**样本
> - 如果全部暴跌，可能是 target model 的 API 出了临时问题（如部署切换、rate limit）
> - 如果部分暴跌，看暴跌样本的特征（task_type、fail_reason），可能有聚类
>
> **方向 C：Selection set 的统计波动**
> - 如果 selection set 很小（如 <20），hard metric 的波动本身就很大
> - 检查 `selection_hard` 和 `selection_soft` 是否一致（如果不一致，可能是 scoring 问题）
>
> **根因定位后**：如果确认是 skill 改坏了，gate 应该 reject，current_skill 不会变。但如果 gate 误判（比如 selection set 太小导致 false accept），需要人工介入回滚到 best_skill。"

**追问应对**：

- **"如果 gate reject 了，但 current_skill 还是变了，可能是什么 bug？"**
  - "看 `current_origin` 字段。如果 `current_origin` 是 'rollback'，说明是手动回滚；如果是 'gate_accept'，说明 gate 逻辑有问题。最常见的 bug 是：`candidate_score > current_score` 的判断用了错误的 metric（比如 soft 比 hard 高，但 gate_metric 是 hard）。检查 `evaluate_gate()` 的调用参数。"

### 3.3 面试官必问：如果系统跑得很慢，怎么排查性能瓶颈？

**推荐回答框架： profiling + 瓶颈定位 + 优化方案**

> "我会看 `step_record.json` 里的 `timing` 字段，看哪个阶段耗时最长：
>
> **Rollout 慢**：
> - 检查 `exec_timeout` 是否设得太大（如 SpreadsheetBench 的 600s）
> - 检查 worker 数是否充分利用了 API 的 TPM（如果 TPM 是瓶颈，加 worker 也没用）
> - 检查是否有大量 timeout（看 `results.jsonl` 的 `fail_reason`）
>
> **Reflect 慢**：
> - minibatch_size 太小 → LLM 调用次数太多
> - analyst_workers 太小 → ThreadPoolExecutor  parallelism 不够
> - 某些 minibatch 的轨迹太长（超过 12KB），导致 LLM 处理慢
>
> **Aggregate 慢**：
> - merge_batch_size 太小 → merge 层级太深
> - LLM merge 调用本身慢（尤其是 patch 数量多的时候）
>
> **Evaluate 慢**：
> - selection set 太大
> - worker 数不够
>
> **通用优化**：
> - 启用 selection cache（相同 skill 不重复 eval）
> - 减少不必要的 merge 层级（如果 patch 数量少，直接一次 merge）"

### 3.4 面试官必问：如果 LLM 在 reflect/merge 阶段输出格式错误（不是合法 JSON），系统怎么处理？

**推荐回答框架：防御性编程 + fallback + 日志**

> "我们在多个层面有防御：
>
> **1. JSON 提取**：`extract_json()` 函数会尝试从 LLM 输出中提取 JSON，即使被 markdown fence 包裹也能处理。
>
> **2. Fallback**：如果 extract 失败，reflect 阶段会返回空 patch（不生成 edit）；merge 阶段会 fallback 到简单 concatenation（把所有 edits 拼在一起）。
>
> **3. 日志**：所有 LLM 的原始输出都保存在 step artifacts 中，方便事后分析。
>
> **4. Retry**：LLM 调用本身有 retry（默认 5 次），如果是 transient error（如 502），会自动重试。
>
> **但有一个隐患**：如果 LLM 输出的是 '看起来合法但实际上语义错误' 的 JSON（比如字段名拼错），我们的 extract 不会报错，但后续代码会 KeyError。这是 LLM-based 系统的通病，只能靠 prompt engineering（明确要求字段名）和 defensive coding（用 `.get()` 而不是 `[]`）来缓解。"

## 4. 工程落地能力

### 4.1 面试官必问：SkillOpt-Lite 在实际部署中会遇到哪些工程挑战？怎么解决？

**推荐回答框架：真实场景 + 技术方案 + 成本考虑**

> "主要挑战和解决方案：
>
> **1. API 成本与速率限制**
> - Challenge：Rollout 阶段大量并发 LLM 调用，容易触发 TPM/ RPM 限流
> - Solution：Dual deployment 允许 target 用便宜模型（如 gpt-5.4-nano），optimizer 用强模型但调用量小；ThreadPoolExecutor 控制并发度；exponential backoff 自动重试
>
> **2. 长时间运行的任务管理**
> - Challenge：SpreadsheetBench 单条任务可能跑 600s，整个 loop 可能跑几小时
> - Solution：Resume 机制（runtime_state.json + history.json）；step artifacts 持久化；coding agent 的 slash-command 驱动，用户不用等
>
> **3. Skill 版本管理**
> - Challenge：训练过程中 skill 不断变化，怎么追踪哪个版本最好
> - Solution：每个 step 保存 `skill_vNNNN.md`；best_skill.md 始终指向最优版本；history.json 记录完整轨迹；HITL 支持人工 review 和 cherry-pick
>
> **4. 多环境适配**
> - Challenge：6 个 benchmark 的 rollout 逻辑完全不同（有的需要 simulator，有的需要代码执行，有的需要图像输入）
> - Solution：EnvAdapter 抽象接口，每个 benchmark 独立实现；prompt 支持 env-specific override
>
> **5. Auth 管理**
> - Challenge：不同团队用不同的 API provider（Azure、OpenAI、Together、vLLM）
> - Solution：三种 auth mode（azure_cli / azure_key / openai），通过 env.sh 和 .env 配置，代码内部统一处理"

**追问应对**：

- **"如果要把 SkillOpt-Lite 部署成 SaaS 服务，你会怎么做？"**
  - "核心改动：把 coding-agent-driven loop 换成 REST API + job queue。用户上传 skill.md 和 data，系统异步跑 loop，通过 webhook 或 websocket 推送进度。最大的挑战是隔离——不同用户的 skill、data、API key 必须隔离。可以用 container per job 或 namespace per user。成本控制方面，需要实现 fair queueing 和 budget limit per user。"

### 4.2 面试官必问：Dual Deployment 的架构是怎么设计的？如果 optimizer 和 target 用同一个模型会怎样？

**推荐回答框架：架构图 + 动机 + 风险分析**

> "Dual Deployment 在 `backend_config.py` 中实现，optimizer 和 target 有完全独立的：
> - `backend`（azure_openai / openai_chat / claude_chat / codex_exec 等）
> - `optimizer_deployment` / `target_deployment`（模型/部署名）
> - `endpoint`、`api_key`、`auth_mode`
>
> 如果两者用同一个模型，技术上完全可行，但有三个问题：
>
> **1. Cost**：Rollout 的调用量远大于 optimizer（比如 batch=40 的 rollout 需要 40 次调用，而 reflect 只需要 8 次），用同一个强模型会很贵。
>
> **2. Self-reinforcement bias**：如果 target 模型和 optimizer 模型相同，optimizer 看到的 '失败' 可能是模型本身能力不足导致的，而不是 skill 的问题。这时候 optimizer 可能会生成 '迁就模型弱点' 的 skill，而不是真正改善 Agent 行为。
>
> **3. Flexibility**：HarnessOpt 场景下 target 是 exec backend（Codex CLI），optimizer 必须是 chat model。如果强制用同一个模型，HarnessOpt 就做不了。"

### 4.3 面试官必问：Resume 机制是怎么实现的？如果在某个 step 中断了，重启后会不会重复计算？

**推荐回答框架：状态持久化 + 幂等设计 + 边界处理**

> "Resume 的核心是 `runtime_state.json`，它记录：
> - `last_completed_step`：最后成功完成的 step
> - `current_skill_path` / `best_skill_path`：当前 skill 和 best skill 的文件路径
> - `current_score` / `best_score` / `best_step`
> - `current_origin` / `best_origin`：skill 的来源（rollout / gate_accept / rollback / slow_update）
> - `lr_scheduler_state`：学习率调度器的状态
>
> 重启时：
> 1. 读取 `runtime_state.json`，恢复所有状态
> 2. 从 `last_completed_step + 1` 开始继续训练
> 3. Selection cache 从 `history.json` 重建（key=skill_hash, value=score）
>
> **幂等性保证**：
> - Rollout 阶段：检查 `results.jsonl` 是否已存在，存在则跳过
> - Minibatch reflect：检查 `patches/minibatch_*.json` 是否已存在，存在则跳过
> - Slow update / meta skill：检查 `slow_done_path` / `meta_skill_done_path`，存在则跳过
>
> **边界情况**：如果中断发生在 `apply_patch` 之后、`evaluate` 之前，restart 时 current_skill 已经被修改了，但 candidate 没有被 gate 接受。这时候 `current_origin` 会告诉我们 skill 是怎么变的——如果是 'gate_reject' 的回滚没完成，我们需要手动把 current_skill 恢复到 previous step 的版本。这个边界情况在代码中通过 `current_origin` 字段追踪，但极端中断（kill -9）可能导致不一致，需要人工检查 `history.json`。"

### 4.4 面试官必问：系统怎么监控训练状态？有哪些指标需要关注？

**推荐回答框架：指标体系 + 监控方式 + 告警逻辑**

> "我们在两个层面监控：
>
> **1. 自动记录的指标（`step_record.json` + `summary.json`）**
> - **Performance**：`rollout_hard/soft`、`selection_hard/soft`、`test_hard/soft`、`delta_hard`
> - **Cost**：per-stage token usage（prompt/completion/total）、`total_wall_time_s`
> - **Efficiency**：per-step timing（rollout_s, reflect_s, aggregate_s, select_s, evaluate_s）
> - **Quality**：`n_edits_proposed`、`n_edits_applied`、`n_edits_skipped`（protected region）
>
> **2. 需要人工关注的信号**
> - **Regression streak**：连续 N 步 gate reject，说明 optimizer 进入了 '错误循环'，可能需要调整 lr 或 skill 结构
> - **Noop streak**：连续 N 步 applied edits = 0，说明 skill 已经 '饱和'，没有可改进的空间
> - **Token explosion**：某一步的 token 用量突然暴增，可能是轨迹过长或 LLM 输出冗长
> - **Failure concentration**：某个 task_type 或 fail_reason 占比突然升高，可能是 env 问题或 skill 引入了新的问题
>
> **告警逻辑**（在 coding-agent loop prompt 中定义）：
> - `regression_streak ≥ 5` → 自动停止
> - `noop_streak ≥ 5` → 自动停止
> - `best_acc ≥ threshold` → 自动停止（目标已达成）"

## 5. 业务与实际场景理解

### 5.1 面试官必问：SkillOpt-Lite 适合什么样的场景？不适合什么场景？

**推荐回答框架：场景匹配 + 边界条件 + 商业价值**

> "**适合的场景**：
>
> 1. **Agent 行为校准**：基础模型已经很强，但需要针对特定工作流优化。比如 SpreadsheetBench 的 Agent 已经会用 pandas，但需要学会 '先确认列名再操作' 这种领域规范。
> 2. **多模型适配**：同一个 Agent 行为需要在多个模型上部署（如 GPT-5.4-nano 和 GPT-5.5），skill 优化比微调更灵活。
> 3. **快速迭代**：业务场景变化快，需要 '今天改 skill，明天上线' 的速度。
> 4. **可解释性要求高**：skill.md 是人类可读的，audit 和 debug 都很方便。
>
> **不适合的场景**：
>
> 1. **基础能力缺失**：如果模型本身不会做数学推理，skill 优化无能为力，需要微调或换模型。
> 2. **需要深度推理链**：skill 是扁平文本，无法表达复杂的程序化推理逻辑。如果 Agent 需要 '先做 A，根据 A 的结果决定是否做 B，再做 C' 这种动态流程，skill 的静态文本很难覆盖，需要优化 harness 代码（HarnessOpt）。
> 3. **大规模个性化**：如果每个用户都需要不同的 skill，skill 优化的 per-user cost 太高。这时候需要更轻量的方法（如 retrieval-augmented skill）。
> 4. **实时性要求极高**：skill 优化需要多次 rollout，每轮几分钟到几小时，无法实时响应。"

**追问应对**：

- **"如果公司有 100 个不同的 Agent 场景，都要跑 SkillOpt-Lite，成本怎么控制？"**
  - "几个策略：第一，shared optimizer——用同一个强模型（如 GPT-5.5）作为 optimizer，同时服务多个 target，因为 optimizer 的调用量远小于 target；第二，transfer learning——从一个场景学到的 skill edit pattern 迁移到相似场景；第三，progressive optimization——先在小数据集上快速验证 edit 质量，再在大数据集上应用。"

### 5.2 面试官必问：从你的经验看，用户（使用优化后 Agent 的人）最关心什么？SkillOpt-Lite 解决了他们的哪些痛点？

**推荐回答框架：用户视角 + 痛点映射 + 价值量化**

> "用户最关心三个东西：**准确率、速度、稳定性**。
>
> **准确率**：SkillOpt-Lite 通过验证门控保证 skill 不会退化，而且 minibatch reflection 能系统性地解决 failure pattern，不是 '头痛医头'。比如 SpreadsheetBench 上，baseline harness 容易犯 'A1-confusion'（假设数据从 A1 开始），优化后的 skill 明确要求 '先确认数据起始位置'，这个 pattern 被系统性修复了。
>
> **速度**：用户不关心 skill 是怎么优化出来的，只关心 '我的 Agent 是不是更准了'。SkillOpt-Lite 的 'vibe-driven' 交互（`/skillopt-loop`）让非技术用户也能触发优化，不需要懂算法。
>
> **稳定性**：验证门控 + resume + rollback 保证训练过程不会因为单次 bad edit 而崩溃。用户可以放心地跑多轮，知道系统会自动保留好的、丢弃坏的。
>
> **但用户也有没被满足的需求**：比如 '为什么我的 Agent 这次失败了？'——当前系统的可解释性只在 skill.md 层面，不能说清 '为什么这个 edit 被选中/被拒绝'。HITL 模式提供了部分答案，但不是默认开启的。"

### 5.3 面试官必问：上线成本有多高？如果资源有限，应该优先优化哪些部分？

**推荐回答框架：成本拆解 + 优先级排序 + ROI 分析**

> "SkillOpt-Lite 的成本主要在三个部分：
>
> **1. Rollout 成本（最大头，~70-80%）**
> - 和 batch_size × accumulation × num_epochs × 单条 rollout 成本成正比
> - SpreadsheetBench 单条 rollout 可能 30 turns + 600s exec，cost 很高
> - **优化优先级：最高**。因为占比最大，优化 rollout 的 ROI 最高。具体手段：降低 batch_size、减少 max_turns、用更便宜的 target 模型、并行度调到 TPM 上限
>
> **2. Reflect + Aggregate 成本（~15-25%）**
> - 和 minibatch_size × num_epochs × LLM 调用成本成正比
> - 优化手段：增大 minibatch_size（减少调用次数）、用更便宜的 optimizer 模型、减少 analyst_workers
>
> **3. Evaluate 成本（~5-10%）**
> - Selection set 评估是每步必做的，但 selection cache 可以避免重复计算
> - 优化手段：减小 selection set、只用 soft metric（不需要跑 full eval）
>
> **资源有限时的优先级**：
> 1. **先优化 target model**：用更便宜的模型跑 rollout，成本降一个数量级
> 2. **再优化 rollout 本身**：减少 max_turns、exec_timeout，跳过已知难的样本
> 3. **最后优化 optimizer**：用便宜模型做 reflect/merge，或者增大 minibatch_size"

### 5.4 面试官必问：如果业务方说 '我们的场景和你的 benchmark 都不一样'，你怎么适配？

**推荐回答框架：通用方法论 + 最小可行适配 + 快速验证**

> "SkillOpt-Lite 的设计哲学是 'Agent-agnostic'——核心 loop 不依赖任何特定 benchmark。适配新场景只需要三步：
>
> **1. 实现 EnvAdapter（~200-500 行代码）**
> - `rollout()`：把 '当前 skill + 任务' 输入 Agent，得到轨迹和分数
> - `reflect()`：通常可以直接用 `run_minibatch_reflect()`，不需要改
> - `get_dataloader()`：提供 train/val/test split
> - `get_task_types()`：返回任务类型列表
>
> **2. 写 YAML config**
> - 继承 `_base_/default.yaml`，覆盖 `env.name`、`skill_init`、`batch_size`、`exec_timeout` 等
>
> **3. 写 slash-command prompt（可选）**
> - 如果要用 coding-agent 驱动，复制 `.github/prompts/skillopt-loop.prompt.md`，修改 allowlist、rollback tag、stopping conditions
>
> **最小验证**：先跑 `--num_epochs 1 --batch_size 5 --eval_limit 5`，确认整个 loop 能跑通、skill 被修改、gate 能决策。如果这个 smoke test 过了，再调参跑完整训练。
>
> 最难的部分通常是 `rollout()`——需要把业务逻辑（如调用内部 API、操作私有数据库）封装成 Agent 能理解的轨迹格式。但只要 rollout 能产生 `{id, hard, soft, n_turns, fail_reason, task_type, extras}`，后续 pipeline 完全通用。"

## 6. 综合模拟面试（附完整回答范例）

### 面试官："介绍一个你最深入的项目，重点讲技术难点和你的贡献。"

**完整回答范例（约 3 分钟）**：

> "我做过一个叫 SkillOpt-Lite 的项目，核心是**不微调模型，而是通过迭代编辑一份 Markdown skill 文档来优化 Agent 行为**。传统方法是 fine-tune 模型权重，但我们的出发点是：很多场景下，Agent 的能力已经够了，缺的是 '操作规范'——比如 '先确认列名再操作 Excel' 这种流程 guidance。修改 skill 比微调更轻、更快、更可解释。
>
> 项目的核心 pipeline 叫 ReflACT，有 6 个 stage：Rollout、Reflect、Aggregate、Select、Update、Evaluate。其中三个技术难点：
>
> **第一，Reflect 阶段怎么让 LLM 稳定地产出有用的 edit？**
> 如果逐条分析轨迹，成本高而且容易学到局部噪声。我们设计了 minibatch reflection——把失败轨迹按 M=8 分组，一次 LLM 调用分析一批， analogous to minibatch SGD。这样既降低成本，又能让 LLM 发现跨样本的共性 failure pattern。但这里有个 trade-off：M 太大，LLM 可能漏掉细节；M 太小，噪声大。我们默认 M=8，并在 prompt 里明确要求 '分析 COMMON patterns，不是 individual edge cases'。
>
> **第二，怎么防止 LLM 的 edit 破坏 skill 的长期结构？**
> LLM 生成的 edit 是非确定性的，可能互相冲突，也可能不断 append 导致 skill 越来越冗长。我们用了两个机制：一是 validation gate，类比 early stopping，每次修改后先在 selection set 上验证，不好就回滚；二是 protected region，skill 里有一个 `<!-- SLOW_UPDATE_START -->` 区块，step-level 的 edit 不能碰，只有 epoch 级的 slow update 能改。这相当于把 skill 分成 '战术层'和 '战略层'。
>
> **第三，系统怎么保证可恢复和可调试？**
> 训练可能跑几小时，中途中断是常态。我们设计了完整的 resume 机制：runtime_state.json 记录最后完成的 step，history.json 记录完整历史，每个 step 的 artifacts（merged_patch、ranked_edits、candidate_skill）都持久化。如果某一步分数暴跌，可以逐 stage 检查 artifacts，定位是 rollout 问题、reflect 问题还是 merge 问题。
>
> 我的主要贡献是实现了整个 training loop 和 minibatch reflection 引擎，以及 6 个 benchmark 的 EnvAdapter。项目在 ICML 2026  poster，在 6 个 benchmark 上显著超越了 SkillOpt baseline。"

### 面试官："你有没有遇到过实验结果和预期不一致的情况？怎么处理的？"

**完整回答范例**：

> "有。SpreadsheetBench 上，我们预期增大 batch_size 会提升效果（因为 minibatch reflection 的 M 更大，能发现更稳定的 pattern），但实际发现 batch_size 从 40 增到 80 后，效果反而下降了。
>
> **排查过程**：
> 1. 先确认 rollout 本身没有问题——用同样的 skill，batch_size=40 和 batch_size=80 的 rollout 分数一致，说明 rollout 阶段无 bug。
> 2. 检查 reflect 阶段的 patch 质量——发现 batch_size=80 时，minibatch 内的轨迹多样性显著增加（因为 SpreadsheetBench 有 cell-level、sheet-level、other 三种任务类型），M=8 的分组可能把不同类型的轨迹混在一起，LLM 分析时 '共性 pattern' 反而被稀释了。
> 3. 验证假设：手动把 batch_size=80 的轨迹按 task_type 分层，再分别做 minibatch reflection，效果回升了。
>
> **解决方案**：
> 我们没有改代码，而是在 config 里加了 `stratified_sampling` 的 option——rollout 时按 task_type 分层采样，确保每个 minibatch 内的轨迹类型一致。但这增加了 rollout 的复杂度，所以最终论文里还是用 batch_size=40，靠随机分组的统计平均来保证质量。
>
> 这次经历让我体会到：**实验预期和实际结果不一致时，不要先想着 '怎么调参让结果变好'，而是要先理解 '为什么和预期不一样'**。有时候预期本身就是错的——batch_size 越大不一定越好，这取决于数据的分布结构。"

### 面试官："如果让你把 SkillOpt-Lite 的产品化，你会怎么设计系统架构？"

**完整回答范例**：

> "我会设计成一个**异步的 skill optimization service**，核心模块：
>
> **1. Job Queue**：用户提交 'optimize my skill' 请求，包含 skill.md、data、config。Job 被放到队列里（如 Redis Queue 或 RabbitMQ）。
>
> **2. Worker Pool**：每个 worker 是一个完整的 ReflACT loop，但做了改造：
> - Rollout 和 Evaluate 阶段改为调用用户的 Agent API（而不是直接跑本地代码），这样支持异构 Agent
> - LLM 调用通过统一的 API gateway，做 rate limit、cost tracking、fallback
>
> **3. Artifact Store**：每个 job 的产物（skill snapshots、step records、patches）存在对象存储（S3/MinIO）里，而不是本地文件系统。
>
> **4. Progress Tracker**：用户可以通过 API 查询 job 状态，或者 webhook 接收通知。关键事件：'round N started'、'round N accepted'、'round N rejected'、'training completed'。
>
> **5. Rollback & Versioning**：每次 accept 都自动打 git tag，用户可以在 UI 上对比不同版本的 skill，一键回滚。
>
> **6. Cost Guard**：每个 job 有 budget limit（如 $100），超过后自动停止并返回当前 best skill。实时跟踪 token 用量，接近 budget 时预警。
>
> **最大的工程挑战**是**隔离性**——不同用户的 API key、skill、data 不能互相干扰。我会用 Kubernetes namespace per user 或者 container per job 来隔离。另一个挑战是**长任务的可靠性**——worker 可能 crash，需要 checkpoint/restart 机制。"
